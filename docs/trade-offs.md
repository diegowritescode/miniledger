# Trade-offs

The senior signal is showing that options were weighed, not just that something was built. Every
load-bearing decision below has a full ADR; this page is the map. Correctness is the product, so
most of these trades buy **provable correctness** at a deliberate, documented cost.

## Decisions made

| Decision                                                                                                                                                                    | Alternatives considered                                                                       | Why this one                                                                                                                                                                                                                                  | Cost accepted                                                                                                                                                                                                                                      |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Money as signed `bigint` minor units** ([ADR-004](adr/004-money-representation.md))                                                                                       | Floating point; Postgres `numeric`; separate debit/credit columns; money strings              | Integer arithmetic is **exact and total**, so sum-zero and reconciliation are provable, not probabilistic; compact, indexable, `CHECK`-friendly                                                                                               | `bigint` ergonomics at JSON/DB boundaries (strings, no `number` coercion); whole-minor-unit only — no fractional cents until FX                                                                                                                    |
| **Sum-zero enforced twice — pure aggregate + deferred DB trigger** ([ADR-005](adr/005-double-entry-model.md))                                                               | Enforce in app code only; a per-row `CHECK`                                                   | The invariant must survive a buggy path **and** a direct SQL write; a single-row `CHECK` cannot span a transaction's legs, so a `CONSTRAINT TRIGGER … DEFERRABLE INITIALLY DEFERRED` (evaluated at COMMIT) is the correct multi-row mechanism | Hand-written trigger SQL is part of the correctness surface (covered by integration tests); every correction is a compensating entry, never an edit                                                                                                |
| **Ordered `SELECT … FOR UPDATE` over SERIALIZABLE** ([ADR-006](adr/006-concurrency-safe-balances.md))                                                                       | `SUM()` on read at READ COMMITTED; SERIALIZABLE everywhere; advisory locks; optimistic CAS    | Locking the materialized balance row in **account-id order** gives no lost updates, no write-skew overdraft, and **no deadlocks** — without retry-on-`40001` machinery; the locked row is also the reconciliation anchor                      | Writes to a hot account (e.g. `@world`) serialize; forgetting the ordered lock on a new write path would reintroduce the hazards (guarded by the concurrency test)                                                                                 |
| **Postgres-authoritative idempotency, key claimed in-transaction** ([ADR-007](adr/007-idempotency.md))                                                                      | Redis for the idempotency record; app-level lock then check-then-act; caching error responses | The key row and the postings must commit **atomically** — Redis cannot; the unique-key `INSERT … ON CONFLICT` self-serializes concurrent duplicates                                                                                           | Stored responses accumulate (a TTL sweep is a later op concern); a failed first attempt re-executes on retry (errors are not cached — a deliberate choice)                                                                                         |
| **Per-account hash chain over a single global chain** ([ADR-008](adr/008-audit-hash-chain.md))                                                                              | One global chain; Merkle tree per period; external WORM log                                   | A per-account chain advances under the balance lock a transfer already holds, so tamper-evidence costs **no concurrency**; a global chain would serialize the whole ledger                                                                    | No single global hash (conservation covers cross-account); **not signed** — proves consistency, not authorship (signing deferred); verification is O(postings) per account                                                                         |
| **Hybrid auth: capability PEP + local ownership, over full ReBAC** ([ADR-009](adr/009-accesscore-integration.md))                                                           | Write owner tuples to AccessCore (full ReBAC); pure-local auth                                | Opening an account stays a **single local transaction** (no two-system commit — the distributed-transaction smell); "who owns account X" is a ledger fact kept in the ledger; still exercises the SDK end-to-end                              | Privileged ops depend on AccessCore availability (mitigated: short timeout, fail-closed 503, local reads); an offline verify honors a revoked-but-unexpired token on local reads (≤ TTL); the token is verified twice on writes (defense-in-depth) |
| **SDK from public npmjs, not GitHub Packages** (AccessCore [ADR-017](https://github.com/diegowritescode/accesscore/blob/main/docs/adr/017-sdk-packaging-and-publishing.md)) | GitHub Packages / a private registry                                                          | A public, tokenless install preserves the **clone-and-run** contract: `npm ci` and `docker compose up` work from a clean clone with no AccessCore credentials                                                                                 | Publishing is a cross-repo AccessCore concern; it must use npm Trusted Publishing (OIDC), as classic/automation tokens are deprecated                                                                                                              |
| **Single-package repo, not a workspace** ([ADR-002](adr/002-repository-shape.md))                                                                                           | Mirror AccessCore's pnpm + Turbo + changesets; a spine monorepo; Nx                           | MiniLedger is one build unit with no shared internal packages, so the workspace tooling would be pure overhead; the high-signal gates (lint, merged coverage, conventional commits, green CI, one-command boot) are kept                      | Intentional inconsistency with AccessCore's shape (documented, not drift); no turbo task cache (negligible for one package)                                                                                                                        |
| **Hexagonal modular monolith + DDD, not microservices** ([ADR-001](adr/001-architecture-style.md))                                                                          | Microservices (accounts + postings); anemic layered CRUD                                      | A transfer is one atomic consistency boundary; keeping `Account` and `JournalTransaction` in one process makes atomicity structural, while pure domain layers make the invariants unit/property-testable                                      | More upfront structure and mapping boilerplate than CRUD; discipline needed to keep `access`/`audit` out of `ledger` internals                                                                                                                     |

## Things deliberately NOT done

- **Holds / two-phase transfers (authorize-then-capture).** Every transfer is single-phase and
  immediate. Reserving funds is a later ring, not a day-one cost.
- **Cross-currency FX.** A transaction fixes one currency ([ADR-004](adr/004-money-representation.md));
  FX will be modeled as balanced per-currency legs through a settlement account, not by summing
  across currencies. Out of scope until then (`numeric` becomes the natural upgrade if fractional
  minor units appear).
- **Domain-event outbox → EventBridge.** The seam is designed for ([ADR-001](adr/001-architecture-style.md))
  but not built; it lands with spine project #3 without reshaping the domain.
- **Cryptographic signing of postings.** The hash chain is tamper-**evident**, not
  non-repudiable; KMS/Vault-held signing keys are the additive upgrade
  ([ADR-008](adr/008-audit-hash-chain.md)), deferred to avoid key management this scale does not need.
- **Full ReBAC ownership in AccessCore.** Account-owner tuples are kept local to avoid a two-system
  commit ([ADR-009](adr/009-accesscore-integration.md)); a later ring, not day one.
- **LIVE public deployment, least-privilege DB role, metrics/tracing.** Documented hardening slices
  — see [deployment.md](deployment.md).

See [`adr/`](adr/) for the full records.
